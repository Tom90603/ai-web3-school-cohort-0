## Agent 付款完整链路

```
用户说："帮我从 API 买一份市场报告"
         │
         ▼
┌─────────────────────────────────────────────┐
│  ① 预算授权（Budget Authorization）            │
│  ─ 用户设预算：「今天最多花 3 USDC」           │
│  ─ 存入 Wallet Policy / Session Key          │
│  ─ ⚠️ 预算要先谈好，不能边花钱边加额度          │
│  ─ 预算要分多层：全局上限 / 单任务上限 /         │
│    单次调用上限 / 服务方上限                    │
└───────────────────┬─────────────────────────┘
                    │
                    ▼
┌─────────────────────────────────────────────┐
│  ② 获取报价（Quote）                           │
│  ─ Agent 向 API 服务商要报价                   │
│  ─ 好的报价包含：价格 + 币种 + 收款地址 +       │
│    有效期 + 交付条件 + 退款条件 + quote_id     │
│  ─ ⚠️ 报价有过期时间！过期就不能再用             │
│  ─ Agent 检查：是否在预算内？服务方可信吗？      │
└───────────────────┬─────────────────────────┘
                    │
                    ▼
┌─────────────────────────────────────────────┐
│  ③ 创建支付意图（Payment Intent）              │
│  ─ 不是真付款，是表达「我想为这个服务付钱」       │
│  ─ 绑定额度、收款方、任务上下文、可接受结果       │
│  ─ ⚠️ Payment Intent 和实际结算分开             │
│    这样审计时能回到「用户当时授权了什么」         │
│  ─ 字段参考：用户目标 / 服务方 / 最大金额 /      │
│    币种 / 链 / 过期时间 / quote引用 / 是否重试  │
└───────────────────┬─────────────────────────┘
                    │
                    ▼
┌─────────────────────────────────────────────┐
│  🔄 预算检查 + Policy 验证                      │
│  ─ 这笔付款是否属于当前任务？                    │
│  ─ 服务方是否在白名单里？                       │
│  ─ 金额是否超过频率/金额上限？                   │
│  ─ 通过 → 进结算环节                           │
│  ─ 不通过 → 拒绝 / 请求人工确认                 │
└───────────────────┬─────────────────────────┘
                    │
          ┌─────────┴──────────┐
          ▼                    ▼
┌──────────────────┐  ┌──────────────────┐
│ ④a 直接付款        │  │ ④b 托管付款(Escrow)│
│ (低风险/小额)       │  │ (高风险/大额/      │
│                    │  │  不信任场景)       │
│ → 直接转账          │  │ → 资金锁定在合约    │
│ → 拿到结果          │  │ → 等交付验收后释放  │
└────────┬─────────┘  └────────┬─────────┘
         │                     │
         ▼                     ▼
┌──────────────────┐  ┌──────────────────────┐
│  ⑤ 服务交付        │  │  ⑤ 服务交付 + 验收    │
│  → 拿到结果         │  │  → 服务方交结果 + proof │
│  → 收据记录         │  │  → Evaluator判断是否合格│
└────────┬─────────┘  │  ├ 通过 → 释放资金      │
         │             │  ├ 超时 → 退款         │
         ▼             │  └ 争议 → Dispute仲裁   │
┌──────────────────┐  └───────────┬──────────┘
│  ⑥ 收据（Receipt） │              │
│  ─ 记录：谁付给谁    │              ▼
│  + 金额 + 任务ID   │  ┌──────────────────┐
│  + 交易哈希 + 结果   │  │  ⑥ 收据（Receipt）   │
│  + 验收状态          │  │  + 交付证明 hash    │
│                    │  │  + 验收记录         │
└────────┬─────────┘  │  + 争议记录（如有）   │
         │             └──────────────────┘
         ▼
┌─────────────────────────────────────────────┐
│  ⑦ 预算更新 + 系统记录                        │
│  ─ 扣除已花金额                               │
│  ─ 记录 quote / receipt / 交易哈希到 trace    │
│  ─ 回应用户：「花 0.3 USDC 买了报告，还剩       │
│    2.7 USDC」                                 │
│  ─ ⚠️ 如果有订阅 → 记录下次扣款时间和剩余额度   │
└─────────────────────────────────────────────┘
```

## 两条路径对比

| | 直接付款 | Escrow 托管付款 |
|:---|:---|:---|
| 适合场景 | 小额、可信服务方、按次API调用 | 大额、不信任服务方、代码/数据交付 |
| 资金流程 | 先付→再拿结果 | 先锁→交付验收→再释放 |
| 风险 | 服务方不交付也没办法 | 失败可退款，有争议路径 |
| 复杂度 | 低 | 高（需要状态机、proof、evaluator） |

## 一次实战走位：Agent 买 API 报告

```
① 用户说"今天最多 3 USDC，帮我买个市场报告"
   → 预算写入 policy

② Agent 找到 API，对方报价：0.3 USDC/份，30秒有效
   → 确认在预算内，服务方可信

③ 创建 Payment Intent：目标=买报告，金额=0.3 USDC，
   收款方=API Co., quote引用=Q-001

④ 预算检查通过 → 因为是小额/可信 → 直接付款
   → 转账 0.3 USDC

⑤ API 返回报告内容 + receipt

⑥ 扣除预算：剩余 2.7 USDC
   系统记一笔：花了 0.3，买了报告，任务完成
```
