# 💰 机器支付与结算链路图

> 产出时间：2026-05-29 | 学员：Tom90603

## Agent 付款完整链路

### 阶段① 预算授权

- **做什么**：用户设定 Agent 今天的最大支出范围
- **关键点**：预算要先谈好，不能边花边加额度
- **预算结构**：全局上限 → 单任务上限 → 单次调用上限 → 服务方上限
- **存储位置**：Wallet Policy 或 Session Key

### 阶段② 获取报价（Quote）

- **做什么**：Agent 向服务方询问价格
- **好报价包含**：价格 + 币种 + 收款地址 + 有效期 + 交付条件 + 退款条件 + quote_id
- **⚠️ 报价有有效期**：过期后价格可能变化，不能用
- **Agent 检查**：是否在预算内？服务方可信吗？

### 阶段③ 创建支付意图（Payment Intent）

- **做什么**：表达"我想为这个服务付钱"，但还没真付
- **为什么要有这一步**：把"用户授权了什么"和"实际结算"分开，方便审计
- **字段参考**：用户目标 / 服务方 / 最大金额 / 币种 / 链 / 过期时间 / quote引用 / 是否允许重试

### 阶段④ 预算检查 + Policy 验证

| 检查项 | 说明 |
|:---|:---|
| 是否属于当前任务？ | 防止 Agent 偏离用户目标花钱 |
| 服务方是否在白名单？ | 防止被恶意引导到未授权服务方 |
| 金额是否超上限？ | 频率限制 + 单次上限 |
| 结果 | 通过 → 进结算；不通过 → 拒绝或请求人工确认 |

### 阶段⑤ 付款（二选一）

#### 路径 A：直接付款（低风险/小额）

- 适合：按次API调用、可信服务方
- 流程：直接转账 → 拿结果

#### 路径 B：Escrow 托管付款（高风险/大额/不信任场景）

- 适合：代码交付、数据报告、跨 Agent 委托
- 流程：资金锁定在合约 → 服务方交付 + 提交交付证明（proof）→ Evaluator 验收

| 验收结果 | 资金去向 |
|:---|:---|
| 通过 | 释放给服务方 |
| 超时/失败 | 退款给用户 |
| 争议 | 进入 Dispute 仲裁 |

### 阶段⑥ 收据（Receipt）

- **记录内容**：谁付给谁 + 金额 + 币种 + 任务ID + 交易哈希 + 结果引用 + 验收状态
- **用途**：后续对账、声誉评分、争议证据
- **对 Agent 经济来说**：收据会成为服务方信誉的输入

### 阶段⑦ 预算更新 + 系统记录

- 扣除已花金额
- 记录 quote / receipt / 交易哈希到 trace
- 回应用户：「花了 0.3 USDC 买了报告，还剩 2.7 USDC」
- 如果是订阅项目：记录下次扣款时间和剩余额度

## 两条路径对比

| 维度 | 直接付款 | Escrow 托管付款 |
|:---|:---|:---|
| 适合场景 | 小额、可信服务方、按次API | 大额、不信任服务方、代码交付 |
| 资金流程 | 先付 → 再拿结果 | 先锁 → 交付验收 → 再释放 |
| 风险 | 服务方不交付也没办法 | 失败可退款，有争议路径 |
| 复杂度 | 低 | 高（状态机 + proof + evaluator） |

---

## 一次实战走位：Agent 买 API 报告

1. **用户设预算**："今天最多 3 USDC，帮我买个市场报告" → 写入 policy
2. **Agent 询价**：API 报价 0.3 USDC/份，30秒内有效 → 确认在预算内
3. **创建支付意图**：目标=买报告，金额=0.3 USDC，收款方=API Co.
4. **预算检查通过** → 小额/可信 → 直接付款，转账 0.3 USDC
5. **API 返回报告** + receipt
6. **预算更新**：剩余 2.7 USDC，系统记录了一笔消费
