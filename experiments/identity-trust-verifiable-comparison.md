# 传统 vs 区块链：身份 / 授权 / 信任 / 可验证性对比

> **Module 4 — Agent 身份与信任 产出**
> 学员：Tom90603 | 方向：产品研究
> 阅读：Agent Identity + Agent Trust & Reputation + Verifiable AI

| 维度 | 传统互联网 | 区块链 / Web3 |
|------|-----------|--------------|
| **🆔 身份** | 平台账号（Google/微信/邮箱）<br>由平台发号、平台存、平台删<br>换个平台从头建信用 | Agent Identity（DID/VC）<br>身份锚定在链上 Registry<br>跨平台可解析、可验证、owner 自己控制 |
| **🔐 授权** | OAuth / Session Token<br>授权粒度粗（全有或全无）<br>平台决定你能干什么 | Smart Account / Session Key<br>可精细控制（仅转账xx额度、仅调用xx合约）<br>可设有效期、可撤销、可审计 |
| **🩺 信任** | 评分系统（五星/好评率）<br>容易被刷、可删评、平台算法黑盒<br>离开平台信任归零 | Reputation + Stake + Attestation<br>行为绑定任务 ID + 链上记录 + 可查证据<br>刷评有成本（Stake 可 Slash） |
| **🔬 可验证性** | 看日志 + 看数据库<br>日志可删、后台可改、证明力弱<br>用户只能"相信平台说" | Verifiable AI（TEE / ZK / Audit Trail）<br>输出可溯源、环境可证明、证据不可篡改<br>用户能验证"这个结果到底怎么来的" |

## 关键词对照

| Web3 概念 | 传统类比 |
|-----------|---------|
| DID（去中心化身份） | 互联网世界没有的——你的身份不属于任何平台 |
| Verifiable Credential | 毕业证，但不用打电话给学校验证，加密签名自证 |
| Session Key | 酒店房卡——有效期到退房，丢了也不怕，主卡锁上就行 |
| Stake | 押金——不过押金是中心化冻着的，Stake 是合约锁着的 |
| Slashing | 押金被扣——但由代码执行，不是房东说了算 |
| TEE | 在隔音玻璃房操作——外面能看到你在干不能看到你在干嘛，但能确认你没作弊 |
| ZK Proof | 你证明自己年龄>18，但不给身份证——不泄露额外信息 |
| Audit Trail | 飞机黑匣子——记录了所有操作，出事可以复盘 |

## PM 视角的关键洞察

### 1. 信任不是分数，是证据集
传统做产品时，我们常问"给用户展示什么分数"。但在 Agent 场景里，**一个数字不如一组可展开的证据链**。产品设计上，要给用户提供"查证路径"而不是"权威标签"。

### 2. 学习曲线 vs 信任体验
DID / VC / ZK / TEE — 这些底层技术用户不需要理解。产品层面要做的是翻译：
- "这个 Agent 通过了能力测试" → 背后是 Attestation
- "这次操作有争议可追溯" → 背后是 Audit Trail
- "这个结果可信" → 背后可能是 TEE 证明

### 3. 身份分拆的好处
传统账号是"登录即信任"——你登录了 Google 就能用所有服务。
Web3 思路相反：**身份 + 授权 + 声誉 + 验证 应该是四个独立模块**。一套身份可以挂多个权限配置，配不同声誉来源，用不同验证方案。用户选择更灵活，但产品设计更复杂。

### 4. 从产品角度看最难的三个问题
1. **验证的成本谁来承担？** ZK 证明贵、TEE 机器贵、Audit Trail 存储贵。产品怎么分层收费？
2. **刷声誉怎么办？** 链上记录不能删，但好评还是可以买。Stake 有成本门槛，小额 Agent 怎么办？
3. **用户怎么理解"可验证"？** 大多数人不看私钥签名，给他们看什么 UI 能既准确又不吓人？
