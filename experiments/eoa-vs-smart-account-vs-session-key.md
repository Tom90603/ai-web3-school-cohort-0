# EOA vs Smart Account vs Session Key 对比

> 共学营 Week 2 ｜ Agent 钱包与权限模块产出
> 方向：产品研究

---

## 概述

AI Agent 要从"回答问题"进化到"执行链上操作"，钱包和权限设计是绕不开的基础设施。本文梳理 EOA、Smart Account、Session Key 三个核心概念的定义、边界、实际产品代表、三者关系，以及各自的优缺点。

---

## 一、定义与边界

### EOA（Externally Owned Account）

**定义：** 区块链上的普通账户，由一对公私钥控制。谁持有私钥，谁就拥有账户的完全控制权。

**边界：**
- 权限模型是二元的：要么完全控制，要么完全不能
- 无法设置规则（限额、白名单、多签）
- 没有恢复机制——私钥丢失即资产丢失
- 只有一个"主人"

### Smart Account

**定义：** 通过智能合约实现的账户，控制权由合约代码定义，而非单一私钥。本质上是"可编程的账户"。

**边界：**
- 权限由合约规则决定（非私钥）
- 可设置多签、限额、白名单、恢复机制
- 可通过模块扩展功能（Session Key、Guard 等）
- 部署和维护有 Gas 成本
- 部分 DApp 不支持智能账户调用

### Session Key

**定义：** 在 Smart Account 框架内签发的一种临时、受限的授权凭证。允许被授权方在规定范围内自动执行操作。

**边界：**
- 必须依附于 Smart Account，不能独立存在
- 有时间限制、额度限制、合约调用限制
- 可随时被创建者撤销
- 仅供低风险自动操作使用

## 二、实际产品代表

### EOA 类产品
| 产品 | 说明 |
|------|------|
| MetaMask | 最流行的浏览器端 EOA 钱包 |
| Rabby | 支持多链的 EOA 钱包，用户体验较好 |
| OKX Wallet | 兼容 EOA 模式和智能账户模式 |
| 各类硬件钱包 | Ledger、Trezor 等，私钥存储在硬件中 |

### Smart Account 类产品
| 产品 | 说明 |
|------|------|
| Safe | 最成熟的智能账户基础设施，多签 + 模块系统，DAOs/团队广泛使用 |
| ZeroDev | 面向开发者的 AA SDK，开箱即用的智能账户 |
| Biconomy | AA 基础设施，提供 Paymaster 代付 Gas 服务 |
| UniPass | 基于邮箱登录的智能账户，内置社交恢复 |
| Rhinestone | 专注于 Agent Wallet 场景的模块化智能账户 |

| 产品 | 说明 |
|------|------|

## 三、三者关系

三者不是替代关系，而是**分层协作**关系。

```
你（主私钥 / EOA）
   │
   │ 部署 / 控制
   ▼
Smart Account（规则引擎 / 权限层）
   │
   ├── 定义规则：谁做什么、何时做、怎么做
   ├── 多签 / 恢复 / Paymaster 等能力
   │
   └── 签发 Session Key ──→ AI Agent
            │                    │
            │ 限时/限额度/限合约  │ 自动执行
            │ 用户可随时撤销      │
            ▼                    ▼
        权限边界              链上操作
```

**典型关系链：**
- 用户通过 **EOA**（或社交登录）控制 **Smart Account**
- Smart Account 定义规则，作为 Agent 的**执行边界**
- **Session Key** 是 Smart Account 签发给 Agent 的**临时授权凭证**
- Agent 持有 Session Key 在限定范围内自动执行

## 四、优缺点对比

| 维度 | EOA | Smart Account | Session Key |
|------|-----|---------------|-------------|
| **本质** | 私钥控制账户 | 合约规则控制账户 | 限时授权凭证 |
| **权限粒度** | 全有或全无 | 可精细配置（限额/白名单/多签/时间） | 在 Smart Account 规则下进一步限制时间、金额、调用目标 |
| **易用性** | 高——用户只需管理私钥/助记词 | 中——需要理解账户概念，部署有操作成本 | 中——需配合 Smart Account 使用 |
| **安全性** | 低——单点失效（私钥丢=全丢，私钥泄露=全被盗） | 高——可设置多签、恢复、Guard 等多层保护 | 中——泄露仅影响有限权限，可快速撤销 |
| **兼容性** | 高——所有 DApp 都支持 EOA 签名 | 中——部分 DApp 不支持合约账户调用 | 低——仅在支持 Session Key 协议的平台上可用 |
| **恢复机制** | 无——私钥丢失不可恢复 | 有——社交恢复、多签恢复、时间锁等 | 无——丢失可撤销，但不可找回 |
| **Gas 成本** | 低——交易即正常 Gas 费 | 较高——部署合约需额外 Gas，模块扩展也需 Gas | 较低——仅首次签发和撤销需 Gas |
| **适用场景** | 个人用户管理个人资产 | DAO/团队金库、高价值账户、Agent 权限底座 | AI Agent 自动执行低风险重复操作 |

## 五、PM 视角总结

三个概念的本质差异可以这样理解：

- **EOA 像现金**——谁拿到就是谁的，没有任何使用记录和限制
- **Smart Account 像公司信用卡**——卡在名下，但额度、可消费类别、审批规则都是预设的
- **Session Key 像酒店房卡**——限时、限区域、限功能，前台（Smart Account）随时可以注销

对于 Agent 场景，最务实的设计路径是：
1. **不要直接把 EOA 私钥交给 Agent**——风险不可控
2. **用 Smart Account 做权限底座**——设定规则框架
3. **用 Session Key 做执行载体**——让 Agent 在受限范围内自动操作

核心原则：Agent（概率系统）可以生成候选动作，但权限边界必须由确定性规则（Smart Account + Guard）来保障。
